IDE · Agent Mode

Switch to Agent Mode — stop using plain chat

Agent mode doesn't just suggest code. It edits files, runs terminal commands, installs packages, and self-corrects errors in a loop — all autonomously.

// One prompt in Agent mode:
> "Add a /health endpoint with tests"

✓ Created src/routes/health.ts
✓ Created tests/health.test.ts
✓ Installed supertest
✓ Ran npm test — 3/3 passed
✓ Fixed lint error in health.ts → re-ran → clean
  • Open Copilot Chat → switch dropdown from Chat to Agent
  • Describe what you want, not how to build it
  • Agent creates files, runs tests, and auto-fixes failures
  • Works in VS Code, VS Code Insiders, and JetBrains IDEs
  • Combine with #file and #codebase for focused context
💡 Pro Tip
Use Agent mode for multi-file tasks: "Refactor the auth module to use JWT and update all 14 test files"

Cloud Agent

Assign issues directly to Copilot

Copilot Cloud Agent runs in the background on GitHub — no IDE needed. Assign an issue and it researches, codes, and opens a PR with CI passing.

📋 Open Issue
→
🤖 Copilot Codes
→
🔀 PR + CI
→
✅ Review & Merge
  • Open any issue → Assignees → select Copilot
  • Agent signs its commits and runs your CI pipeline
  • Works entirely in the cloud — zero IDE time
  • Iterates on PR review feedback automatically
  • Add copilot:agent label to auto-assign future issues
💡 Example Issue
"Fix the 404 error on /api/users/:id when user has no profile photo. Return default avatar URL."
$ gh issue edit 123 --add-assignee @copilot
✓ Copilot assigned to issue #123
✓ PR #124 opened — 3 files changed, CI green

Config

Add custom instructions — once

One file in your repo tells Copilot your coding standards. Every session reads it automatically — IDE, Cloud Agent, CLI, and Code Review.

.github/copilot-instructions.md
.github/instructions/*.instructions.md
AGENTS.md
📄 Sample copilot-instructions.md
- Use TypeScript strict mode for all new files
- Follow the repository pattern: controller → service → repo
- Always add JSDoc comments for public methods
- Use Vitest for testing, aim for 80% coverage
- Never use `any` type — use `unknown` instead
  • Define language, framework, and testing conventions
  • Path-specific instructions with glob patterns
  • Org-level instructions now GA (April 2026)
  • Works across IDE, Cloud Agent, CLI, and Code Review
  • Stack multiple instruction files for different areas of your codebase

Model Selection

Pick the right model for the job

Click the model picker in VS Code or on GitHub. Match the model to the task — each has different strengths and cost multipliers.

🏗️ Complex / Reasoning
  • Claude Opus 4.7 — deep refactors, architecture
  • GPT-5.4 — large multi-file changes
  • o3 — complex reasoning tasks
⚡ Fast / Everyday
  • GPT-5.4 mini — fast completions
  • Claude Sonnet 4.5 — balanced
  • Gemini Flash — high volume, cheap
  • Switch models mid-conversation anytime in chat
  • Each model has a premium request multiplier (1x–50x)
  • CLI supports BYOK & local models via Ollama
  • Cloud Agent defaults to the best available model
💡 Strategy
Quick fixes → GPT-5.4 mini (1x)  |  Architecture → Claude Opus (50x)  |  Bulk tests → Gemini Flash (0.25x)

Prompt Files

Create reusable prompt files

Store task-specific prompts as Markdown files. Invoke them with / slash command in chat. Shareable across your entire team for consistency.

.github/prompts/code-review.prompt.md
.github/prompts/add-tests.prompt.md
.github/prompts/refactor-to-pattern.prompt.md
.github/prompts/deploy-checklist.prompt.md
📄 Sample add-tests.prompt.md
Write unit tests for the selected code using Vitest.
- Cover happy path, edge cases, and error scenarios
- Mock external dependencies with vi.mock()
- Use descriptive test names: "should [verb] when [condition]"
- Aim for >90% branch coverage
  • One file per workflow: review, deploy, migrate, document
  • Include variables, constraints, and output format
  • Team members use the same prompts for consistent output
  • Supports template variables like {{LANGUAGE}}
  • Works in IDE and with Cloud Agent

MCP Servers

Connect external tools via MCP

Model Context Protocol lets Copilot talk to databases, APIs, Docker, Azure, and 50+ integrations mid-conversation. No more copy-pasting context.

💻 Copilot
↔
🔌 MCP
↔
🗄️ DBs · APIs · Cloud
📄 .vscode/mcp.json
{
  "servers": {
    "github": { "command": "npx", "args": ["github-mcp-server"] },
    "postgres": { "command": "npx", "args": ["@modelcontextprotocol/server-postgres"] }
  }
}
  • Copilot reads real data from your systems at runtime
  • Org admins can manage MCP allow-lists for security
  • Works in VS Code, CLI, and Cloud Agent
  • GitHub MCP Server: query issues, PRs, repos, and more
  • 50+ community servers: Figma, Slack, Jira, Docker, etc.

CLI

Copilot CLI — AI in your terminal

Run copilot from any terminal. It reads your repo, makes changes, runs commands, opens PRs — all with your approval before each step.

$ copilot
> "Add rate limiting to all API routes and open a PR"

✓ Analyzed 12 route files
✓ Created src/middleware/rate-limit.ts
✓ Updated 12 route files with middleware
✓ Ran npm test — 47/47 passed
✓ Opened PR #42 — "Add rate limiting middleware"
  • Interactive mode + programmatic mode with -p "prompt"
  • Plan mode: Shift+Tab to preview before execution
  • Supports BYOK and local models (Ollama, LM Studio)
  • MCP servers work in CLI too — same .vscode/mcp.json
  • Install: gh extension install github/gh-copilot

IDE Shortcuts

Master @workspace, @terminal, and slash commands

Give Copilot full repo context with @workspace. Use @terminal for CLI debug. Fire off instant actions with slash commands.

@workspace "how does auth work in this repo?"
@terminal "why did that test fail?"
/fix → auto-fixes selected code
/tests → generates unit tests for selection
/explain → explains code in plain English
/doc → adds JSDoc / docstring
  • #file — attach specific files as context
  • #selection — reference your current editor selection
  • #codebase — search the entire project for relevant context
  • Combine: @workspace #file:schema.ts "add validation for the User model"
💡 Power Combo
Select a function → press Cmd+I → type "/tests" → instant test file with edge cases

Code Review

Let Copilot review your pull requests

Copilot Code Review catches bugs, security issues, and style violations before human reviewers even look. Enable it org-wide for automatic reviews on every PR.

📤 Push PR
→
🤖 Copilot Reviews
→
🔧 Suggested Fixes
→
✅ Merge
  • Catches security vulnerabilities automatically (SQL injection, XSS, etc.)
  • Suggests concrete code fixes, not just warnings
  • Customize rules with .github/copilot-instructions.md
  • Enable automatic reviews for all PRs in repo settings
  • Review comments are actionable — click to apply fixes
💡 Auto-Review Setup
Repo Settings → Copilot → Code Review → Enable "Automatic reviews" → Choose triggers (all PRs / labeled / draft)

Best Practice

Your repo IS Copilot's brain

Copilot's output quality is directly proportional to your repo quality. Clean structure, good README, consistent patterns, and clear naming — these aren't optional.

❌ Messy Repo
  • No README or docs
  • Inconsistent naming (camelCase + snake_case)
  • Flat file structure, no modules
  • No tests, no types
  • → Weak, generic suggestions
✅ Clean Repo
  • Detailed README.md with architecture
  • Consistent naming conventions
  • Logical folder structure + modules
  • Full test suite + TypeScript strict
  • → Precise, contextual suggestions
🧠 Repo as Context: What Copilot Reads
README.md → project purpose, architecture
.github/copilot-instructions.md → coding standards
AGENTS.md → agent behavior rules
tsconfig.json / .eslintrc → language conventions
Existing code patterns → naming, structure, style
  • Write a solid README — it's the first thing Copilot reads
  • Use Copilot Spaces to share curated context across teams
  • Add architecture decision records (ADRs) to guide long-term patterns
