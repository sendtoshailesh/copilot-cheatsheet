Modes · Strategy

Master 3 modes — Ask, Plan, Agent

Most developers stay in one mode. Power users switch between all three depending on the task — plus unlock Autopilot and /research as power features on top.

💬
Ask
Quick Q&A, explain code, rubber-duck debugging — no file edits
/ask or Shift+Tab
📝
Plan
Asks questions first, writes plan.md, critic agent reviews before coding
--plan or Shift+Tab
🤖
Agent
Autonomous IDE edits, multi-file changes, self-corrects in a loop
Default in VS Code
  • Shift+Tab cycles Ask → Plan → Autopilot in the CLI
  • Ask — quick lookups, explain code, off-record questions that don’t pollute context
  • Plan — use before complex work; the critic agent catches design flaws early
  • Agent — hands-on coding with iterative self-correction across files
  • Autopilot — approve a plan, then it builds unattended (5-message default)
  • /research — spawns multi-agent investigation with read-only Markdown reports

Cloud Agent · CI/CD

Assign issues to Copilot — it ships PRs autonomously

Copilot Cloud Agent runs on GitHub — no IDE needed. It researches, codes, opens PRs, runs your CI pipeline, iterates on review feedback, and resolves merge conflicts.

🔄 Automated Pipeline
  • 📋 Issue created
  • 🤖 Copilot assigned → codes PR
  • ⚙️ Actions CI: lint, test, build
  • 🔀 Auto-deploy to staging
  • ✅ Human review → merge → prod
🛠️ What Agent Handles
  • Opens PRs with passing CI
  • Resolves merge conflicts (3 clicks)
  • Iterates on review feedback
  • Triggers workflow automations
  • Signs commits cryptographically
  • Open any issue → Assignees → select Copilot
  • Runs your existing GitHub Actions workflows (lint, test, build, deploy)
  • Manage sessions from Issues, Projects, or GitHub Mobile
  • Add copilot:agent label to auto-assign future issues

Config · applyTo

Layer custom instructions — org, repo, path, file

Instructions cascade from org → repo → path-specific → inline. Use applyTo glob patterns to scope rules to exactly the right files.

🏢 Org
→
📁 Repo
→
📂 Path
→
🤖 Agent
Level 1: Org-level — enforced for all repos (GA)
Level 2: .github/copilot-instructions.md — repo-wide
Level 3: .github/instructions/*.instructions.md — with applyTo globs
Level 4: AGENTS.md / .agent.md — agent-specific rules
📄 applyTo Pattern Examples
applyTo: "**/*.test.ts"  → testing conventions
applyTo: "src/api/**"   → API design rules
applyTo: "**/*.md"     → writing style guide
applyTo: "infra/**"    → IaC / Terraform rules
  • Instructions stack: org + repo + path-specific all apply together
  • applyTo globs target rules to specific folders, extensions, or patterns
  • Works across IDE, Cloud Agent, CLI, and Code Review
  • Use AGENTS.md to scope per agent; .agent.md for custom agent definitions
  • Org-level instructions are GA — enforced for every developer

📄 Sample add-tests.prompt.md
Write unit tests for the selected code using Vitest.
- Cover happy path, edge cases, and error scenarios
- Mock external dependencies with vi.mock()
- Use descriptive test names: "should [verb] when [condition]"
- Aim for >90% branch coverage
  • One file per workflow: review, deploy, migrate, document
  • Include variables, constraints, and output format
  • Team members use the same prompts for consistent output
  • Supports template variables like {{LANGUAGE}}
  • Works in IDE and with Cloud Agent

Best Practice

Your repo IS Copilot's brain

Copilot's output quality is directly proportional to your repo quality. Clean structure, good README, consistent patterns, and clear naming — these aren't optional.

❌ Messy Repo
  • No README or docs
  • Inconsistent naming (camelCase + snake_case)
  • Flat file structure, no modules
  • No tests, no types
  • → Weak, generic suggestions
✅ Clean Repo
  • Detailed README.md with architecture
  • Consistent naming conventions
  • Logical folder structure + modules
  • Full test suite + TypeScript strict
  • → Precise, contextual suggestions
🧠 Repo as Context: What Copilot Reads
README.md → project purpose, architecture
.github/copilot-instructions.md → coding standards
AGENTS.md → agent behavior rules
tsconfig.json / .eslintrc → language conventions
Existing code patterns → naming, structure, style
  • Write a solid README — it's the first thing Copilot reads
  • Use Copilot Spaces to share curated context across teams
  • Add architecture decision records (ADRs) to guide long-term patterns

/research — deep investigation mode

When normal chat is too shallow, /research launches multiple subagents, gathers evidence, and returns a Markdown report before you touch implementation.

🔬 /research
→
4 Sub-agents
→
📝 MD Report
→
🌐 /share html
$ /research "How does auth work in this repo?"

✓ Spawned 4 research agents
✓ Wrote research/auth-flow.md
✓ Added citations + Mermaid diagram
✓ Logged assumptions and confidence levels
✓ /share html research ready to export
  • Spawns multi-subagent research jobs with Markdown reports + Mermaid diagrams
  • Export the findings as HTML with /share html research
  • Autonomous: never asks questions, makes assumptions, documents confidence
  • Read-only: investigates deeply but never edits files
  • Use it before planning to gather intelligence first
  • Trigger format: /research TOPIC
